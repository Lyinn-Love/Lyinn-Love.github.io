AI 辅助 FPGA 开发：我手搓了一个 Vivado MCP Server

2026-04-05

背景与缘起

今天心血来潮，想要在 Trae IDE 里用大模型（MCP）直接控制 Vivado 进行 FPGA 开发。原本试着使用开源社区或第三方的 mcp_synthpilot，但发现其核心是一个编译好的闭源二进制黑盒，并且存在 License 限制，无法直接修改和扩展功能。

于是，在我的专属 AI 助理（一只贴心的猫娘 🐱）的陪伴下，我们决定从零开始，自己用 Python 手搓一个轻量级的平替版 MCP Server：MiniSynthPilot！

核心实现：TCP 通信与 Tcl 交互

我们知道，Vivado 提供了非常强大的 Tcl 交互接口。如果使用 batch 模式，每次调用都会启动和关闭 Vivado，不仅慢，而且无法直观地看到工程界面的变化。

因此，我们的思路是：

  1. 启动常驻的 Vivado GUI：用 Python subprocess 拉起一个 Vivado 图形界面，并让它运行一个极其轻量的 Tcl TCP Server。
  2. 监听 9999 端口：这个 Tcl Server 在后台默默监听，接收来自 Python 的指令。
  3. MCP 工具封装：使用 FastMCP 库，把“创建工程”、“添加文件”、“综合”、“仿真”等常用操作封装成工具，通过 Socket 将具体的 Tcl 脚本发送给 Vivado 执行。

这样，当我向大模型下达指令时，我的屏幕上就会直接弹出 Vivado 界面，并自动执行诸如建工程、加代码、跑波形等操作，一切尽在掌握！

踩坑记录：Windows 下的编码乱码

在开发过程中，我们遇到了一个非常典型的坑。

当 AI 调用 MCP 生成带中文输出（比如 puts "✅ 实现完成！"）的 Tcl 脚本并通过 Socket 发送给 Vivado 时，Vivado 的 Tcl Console 里出现了一堆“火星文”乱码。

原因分析： Python 脚本在写入临时 .tcl 文件时，默认使用了 UTF-8 编码。然而，运行在 Windows 环境下的 Vivado，其 Tcl Console 默认使用的是系统区域代码页（通常是 GBK 编码）。

解决方案： 我们修改了临时文件的写入逻辑，动态获取系统默认编码来保存 Tcl 脚本：

import sys
import tempfile

sys_encoding = sys.getdefaultencoding() # 通常在中文 Windows 下为 gbk
with tempfile.NamedTemporaryFile(mode='w', suffix='.tcl', delete=False, encoding=sys_encoding) as f:
    f.write(tcl_content)
    tcl_path = f.name.replace('\\', '/')

修改后，Vivado 里的中文输出瞬间恢复了正常，赏心悦目！

成果展示与 Skill 封装

最终，我们成功封装了以下几个非常好用的 MCP 工具：

  • create_project: 一键创建工程，并强制校验器件型号（如 xc7z020clg400-1）。
  • add_source_file: 添加 Verilog 代码并更新层级。
  • run_simulation: 添加 Testbench，设为顶层，并启动行为级仿真。
  • run_implementation: 跑完综合与实现，并自动打开直观的 Device 器件视图！
  • get_synthesis_warnings: 自动解析 runme.log，提取警告信息反馈给大模型进行错误排查。

为了让这套工作流固化下来，我们最后使用 Trae 的 Skill 功能，将整个 Vivado 开发的上下文、避坑指南和交互风格封装成了名为 fpga-vivado 的技能包 (.trae/skills/fpga-vivado/SKILL.md)。

以后只要我想做 FPGA 项目，打开新对话就能直接进入“人机结对编程 + 自动控制 Vivado”的沉浸式体验。

今天也是充满收获的一天，下次见啦！！！
